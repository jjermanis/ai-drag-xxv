version 15	13 minutes dev	0 minutes code gen	6 minutes test		3 hours 53

Notes v15: moved from 20 to 40 frames per second. I am VERY happy with how this looks. It is
the difference maker: looks very smooth now. The basis was change of MS per move from 50 to 25. 
To adjust for more frequent moves, dropped the increment on velocity in half. Finally,
increased the "MPH calculation" to double. This works well. Since this change was made, I
doubled the rate of acceleration checks. Went from checking every 200ms to 100ms. Increment
in velocity got halved AGAIN. Finally, I used Math.floor() on the display speed, since you
can get decimal values here. Average speeds in the races remain the same, with smoothness for
both accelerations and animation.


version 16	35 minutes dev	1 minute code gen	10 minutes test		4 hours 39

Notes v16: the reindeer legs now move to provide a better illustration. This ended up being
challenging. Chat offered to move the legs for racing - I accepted one option it said it
could do. The app now keeps track of a timer.  It would then, for each reindeer crew, adjust the

effort and testing to get the leg movement to look it best. Lots of manual work was needed, 
though I give Chat credit for creating code that was easy to tweak to change the UI.


version 17	20 minutes dev	0 minute code gen	6 minutes test		5 hours 5

Notes v17: this relates directly to the previous change. In v16, it made the reindeer legs run.
As part of that, Chat created an animTick counter to help with the leg steps. Was it necessary? 
Well, it wasn't obvious that it was needed. We'll try without.
I manually removed all the references to animTick and ran it. IT DID work. But the leg movement
was very slow. That's because it was based off of the speed. Therefore, if the speed did not 
change, the legs did not move. It means not only is it slow, but the movement would be linear. 
Definitely not good. Basing leg movement on their X-position ought to work. As the reindeer 
goes faster on the X-axis, the leg speed would increase.
I tried that. It was close, but also looked odd. Easy to explain: each leg was in a different
place. That meant that each leg was at a different phase. It looks odd.
The final implementation was to base it on the x-position of the sleigh. THAT would be the same
for each reindeer. Not only did it finally indicate that animTick wasn't needed... but does 
actually look a bit better without it.

version 21	18 minutes dev	2 minutes code gen	11 minutes test		6 hours 29

For multiple reasons, I am adding a significant new feature. It is to allow both two-player mode
(the original) and one-player. This allows someone to play using the obvious case: sometimes
there is only one available to play. Also, two-player mode is not likely to work on a tablet.
On my iOS, it does not allow multiple clicks - there would prevent a race from being start well.
Single player would be an option. 
The documentation for this is fairly extensive:
* Two new buttons to appear toward the lower-left corner of the game. 
* The two new buttons are vertically aligned with the Reset Race and Clear Records buttons. 
* The first button reads 1P. The second button reads 2P. 
* The default state when then game starts is 2P. The game in 2P mode is exactly like it is now. 
* When the 1P button is pressed, Player 2 (Nitro Nicolas) does not have controls for a person to play. 
	When the 1P button is pressed, the 1st player is playing against the CPU. The green 
	"Nitro Nicolas (P)" button towards the bottom is invisible. The button will not work. 
	The P key will do nothing. 
* When the race starts in 1P mode, the green player Nitro Nicolas starts 500ms after the green 
	light starts the race. 
* When the 2P button is pressed, it goes to the original mode, exactly like before. 
It took several iterations to get this to work.
#1 Was a massive mistake. It change massively. Originally version was 23k; the new version was
6k. Obviously, multiple UI issues arose. I stopped immediately and tried again.
#2 it provided the changes to make. That did not work. One of the changes referred to variables
that did not exist. I did not want to play a guessing game. I sent the existing code and asked
Chat to try again.
#3 made the same mistake as #1 - the generated version was tiny. I asked it to try again. Not
to send the whole updated program (that tremendously fails) and to put the deltas with clear
details. This was much better. It had one line that was not clear - I asked Chat for details.
That all... worked! To be clear there are several observed defects and changes that I see. But
the game is playable in this mode.
I will document it below, but these are the errors I saw:
* The 1P/2P buttons are plain. Need more detail.
* The Reset Race and Clear Record went to right margin. They should have remainder center.
* Random variability should be used for Nicloas in 1P mode. Base it on typical human resources.
